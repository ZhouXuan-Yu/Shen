# AI助手行为规范

## 文档创建原则

### ❌ 不要频繁创建文档

- **禁止**为每个小修改、bug修复创建总结文档
- **禁止**为临时性问题创建单独的MD文件
- **禁止**创建重复性质的文档（如多个"修复总结"、"问题分析"）
- **禁止**在根目录创建临时性的分析文档

### ✅ 仅在以下情况创建文档

1. **重大功能开发**
   - 新模块/新功能的完整开发计划（>20小时工作量）
   - 架构重大变更的设计文档

2. **用户明确要求**
   - 用户明确说"创建一个文档"、"写一份计划"
   - 用户需要可交付的文档成果

3. **长期参考价值**
   - 部署指南、使用手册
   - API文档、开发规范
   - 不会很快过时的技术文档

### 📁 文档存放位置规范

- **根目录**: 仅保留`README.md`和`README_开发指南.md`
- **docs/**: 存放所有长期文档
- **docs/archived/**: 存放历史文档
- **临时文档**: 直接在对话中说明，不创建文件

### 💬 优先使用对话方式

对于以下内容，直接在对话中回复，不创建文档：

- Bug分析和修复说明
- 参数调优建议
- 代码审查意见
- 问题排查过程
- 临时性的计划和建议

## 工作方式

### 优先行动，少说废话

- 用户提出问题 → 直接修复代码
- 用户要求功能 → 直接开发
- 不需要每次都问"是否开始？"、"准备好了吗？"

### 避免过度总结

- 完成任务后，简洁说明即可
- 不需要emoji过多的华丽总结
- 不需要重复已经做过的事情

### 代码优先

- 多写代码，少写文档
- 用代码和注释说话
- 重要的架构决策才需要文档

## 示例

### ❌ 错误做法

```
用户: 修复一个bug
AI:
1. 修复代码
2. 创建"bug修复总结.md"
3. 创建"问题分析.md"
4. 创建"下一步计划.md"
```

### ✅ 正确做法

```
用户: 修复一个bug
AI:
1. 修复代码
2. 在对话中简要说明修复内容
3. 完成
```

### ✅ 可以创建文档的情况

```
用户: 我需要一份团课照片对比功能的完整开发计划
AI:
1. 创建"团课照片对比功能_开发计划.md"（因为是重大功能，用户明确要求）
2. 文件放在docs/目录
```
