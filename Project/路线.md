## HandTalk AI 项目路线图（由 AI 维护）

> 本文件用于记录整个项目的详细规划与阶段进度，**每次架构或实现方案有调整时，都需要同步更新此文件**，作为后续所有开发任务的参考。

---

## 一、当前整体目标

- **后端**：将 `Project/Back/model/last_checkpoint.pt` 集成到 FastAPI 后端，提供稳定的手语识别服务（图片 / 视频 / 后续实时）。
- **前端**：在 `Project/Front` 中打通上传翻译与实时翻译页面，让用户可以从浏览器端上传视频/图片或通过摄像头实时识别，并看到来自后端真实模型的结果。
- **联调**：完成从前端操作 → 后端推理 → 前端展示结果的完整闭环，替换掉现有的“本地模拟结果”逻辑。

---

## 二、阶段规划总览

1. **阶段 A：现状梳理与接口对齐**
   - 明确后端现有识别接口：`/api/v1/recognize/image`、`/api/v1/recognize/video`、`/api/v1/recognize/status` 等。
   - 对齐前端接口文档 `Front/接口.md` 中约定的上传接口：`POST /api/v1/recognize/upload` 及统一响应格式 `ApiResponse<T>`。
   - 梳理 AAA 目录中已验证的推理脚本（如 `demo_infer.py`、`debug_infer_on_train_feats.py`），确认 `last_checkpoint.pt` 的加载方式与推理流程。

2. **阶段 B：后端模型集成（CTC 推理模块）**
   - 在 `Project/Back/app` 下创建专门的 CTC 推理模块（例如：`ctc_recognizer.py`，实际名称可根据现有结构调整）。
   - 负责：
     - 加载 `model/last_checkpoint.pt`（CTC 模型、字符表等）。
     - 将视频/图片数据转为模型需要的特征表示。
     - 执行 CTC 推理与解码，输出文字与置信度等信息。
   - 对外暴露统一函数，例如：
     - `recognize_video_file(path) -> { text, confidence, ... }`
     - `recognize_image_file(path) -> { text, confidence, ... }`（视需要）。

3. **阶段 C：后端 API 层改造 / 补全**
   - 在 `app/main.py` 中接入新的 CTC 推理模块，实现（或改造）以下接口：
     - `POST /api/v1/recognize/upload`：`multipart/form-data` 上传视频/图片，根据文件类型调用对应推理函数。
     - （可选）`POST /api/v1/recognize/video`、`POST /api/v1/recognize/image`：从“模拟结果”改为真实调用模型。
   - 返回结构严格对齐 `Front/接口.md` 中的定义：
     - 统一封装为 `ApiResponse<T>`，并按需构造 `UploadRecognitionResponse`、`RecognitionResult` 等字段。
   - 保留 `/api/v1/recognize/status`、`/api/v1/recognize/health` 等用于前端或调试使用。

4. **阶段 D：前端上传翻译页面接入后端**
   - 在前端（如：`pages/translate.vue`、相关 store）中：
     - 实现文件选择与进度反馈。
     - 使用 `FormData` 调用后端 `POST /api/v1/recognize/upload` 接口。
     - 将返回的 `results`、`videoDuration`、`processedFrames` 等信息在页面上以友好方式展示。
   - 如有历史记录 / 收藏等模块，预留对接接口的扩展点（暂可不实现完整业务逻辑）。

5. **阶段 E：前端实时翻译页面改造**
   - **第一步：HTTP 轮询模式（快速打通）**
     - 在 `pages/recognize.vue` 中，去掉本地 `simulateRecognition()`，改为：
       - 每隔固定时间从摄像头抓取一帧（图片）。
       - 将这一帧作为图片上传到 `/api/v1/recognize/upload`（或专门的 `/recognize/image`），获取识别结果。
       - 将结果写入 `recognitionStore`，驱动现有 UI（文字、拼音、释义、置信度条等）更新。
   - **第二步（可选进阶）：WebSocket 实时识别**
     - 后端实现 WebSocket `/recognize` 端点，处理 `type: 'frame'` 的消息，实时推理并返回 `type: 'result'` 消息。
     - 前端基于已有的 `composables/useWebSocket.ts` 中的 `useRecognitionWebSocket`，持续发送帧数据并展示返回的实时结果。

6. **阶段 F：联调与优化**
   - 在本地同时启动后端与前端，按以下顺序验证：
     1. `/api/v1/recognize/status` 与健康检查接口正常工作。
     2. 通过 Postman 或 curl 测试 `/api/v1/recognize/upload`，确保模型推理结果合理。
     3. 在前端上传页面上传实际视频/图片，确认能获得与后端一致的结果。
     4. 在实时页面进行简单测试，观察延迟与稳定性。
   - 根据测试结果：
     - 优化推理速度（必要时压缩帧率 / 分辨率）。
     - 改进前端交互体验（loading 状态、错误提示等）。
