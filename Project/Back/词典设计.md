
整体目标与约束
核心目标：在现有「手语词典 + 手语识别」项目上，增加一套基于 CE-CSL 数据集的「文本 → 视频推荐」检索系统，支持：
输入自然语言句子，检索出最相似的标注句子，并推荐对应视频；
页面初始展示随机视频样本；
在前端展示相似度/匹配度评分，并与现有词典搜索良好融合。
前提约束：
后端继续使用 FastAPI，端口以 9000 为主（已在 启动.md 和 nuxt.config.ts 中约定）；
前端使用 Nuxt 3 + Pinia，继续沿用现有 runtimeConfig.public.apiBase 方式访问后端；
索引 / 检索优先本地文件 + 内存向量，不依赖云向量库。
一、整体方案设计（按模块拆分）
1. 数据与离线向量索引层
数据源统一：
使用 dataset\CE-CSL\CE-CSL\video\dev 和对应的 dev.csv，抽取字段（如 video_id, video_path, sentence）。
向量模型：
选用一个中文句向量模型（如 bge-m3 或 text2vec-large-chinese，根据你本地环境选一个即可）；
将每条句子编码成一个固定维度的向量。
索引构建脚本：
新建 scripts/build_cecsl_index.py：
读取 dev.csv；
对每行句子做清洗（去空白、统一简体/标点等）；
用向量模型编码为 embedding；
保存为：np.ndarray（所有向量） + metadata.json（每条的 video_id, video_path, sentence 等）。
后续如需加速，可引入 faiss（AnnIndex），但第一版可用纯 numpy + 余弦相似度。
2. 后端检索与推荐 API 层（FastAPI）
索引加载模块：
在 app/ 下新建 rag_index.py（或类似）：
启动时加载 embeddings.npy + metadata.json 到内存；
提供函数：
search(query: str, top_k: int) -> List[Result]
get_random(n: int) -> List[Result]
检索 API：
在 app/main.py 中挂载新路由前缀：/api/v1/video_rag：
GET /api/v1/video_rag/search?query=...&top_k=10
编码 query → 向量；
计算与所有索引向量的余弦相似度；
取 top_k，返回：
videoId, videoPath, sentence, similarity(0-1), 可选 rank。
GET /api/v1/video_rag/random?limit=10
从 metadata 中随机采样 limit 条，附带一个“伪 similarity=1.0”或不带。
RAG 增强（可选扩展）：
后续可增加：
POST /api/v1/video_rag/explain：根据检索结果用 LLM 生成更自然的文本解释或学习提示；
第一版可以先不做，只保留纯检索。
3. 前端数据层（Pinia Store）
新建或扩展 Store：
新建 stores/videoRag.ts，或在 dictionary.ts 中增加一个子模块（推荐新建，职责清晰）；
状态：
results: 检索结果数组；
randomSamples: 初始推荐数组；
query: 当前检索文本；
loading, error, stats（例如本次检索用时等）。
方法：
searchVideos(query: string, topK: number) → 调用 /video_rag/search；
fetchRandomSamples(limit: number) → 调用 /video_rag/random。
4. 前端 UI 与交互（Nuxt 页面）
页面位置：
基于现有 pages/dictionary.vue 扩展右侧区域，或者新开 pages/video-rag.vue 再从词典页跳转；
推荐做法：保持单页体验，在词典页右侧增加「示例视频推荐」区域。
布局方案（示例）：
左侧：现有词典搜索 + 列表；
右侧：
顶部：一个独立的「句子检索」输入框（与词典搜索区分开）；
中部：当前检索结果的视频列表（带播放器缩略图、句子文本、相似度条）；
底部（或无检索时）：随机推荐列表。
交互逻辑：
页面 onMounted：
调用 fetchRandomSamples(limit=10)，展示随机视频；
用户在「句子检索」框中输入文本并回车：
调用 searchVideos；
若有结果：用检索结果替换右侧列表，显示“共 X 条，高匹配/中匹配标签”等；
若无结果：给出友好提示，并可保留随机区块。
相似度展示：
显示百分比（(similarity * 100).toFixed(0) + '%'）；
基于阈值用颜色/标签表示：高匹配 / 中匹配 / 低匹配。
5. 与词典的联动与扩展（第二阶段）
联动一：从词条跳转到例句视频：
在词典词条详情中，增加「查看相关例句视频」按钮；
按词条中文/含义发起 /video_rag/search，展示结果。
联动二：基于类别过滤（可选）：
在检索接口中增加 category 参数（如果 dev.csv 中有类别/主题），在查询时附加过滤；
后续扩展：
增加学习模式（连播检索结果视频，自动循环播放）；
为每个视频增加「收藏/标记难度」等学习功能。
二、按方案分步骤实施（执行顺序）
下面是建议的实施顺序，你可以逐步推进，每一步完成后我们再细化下一步的实现细节和代码。
步骤 1：环境与配置统一（你已经基本完成）
目标：确保前后端端口与 API 前缀一致。
关键动作（已部分完成）：
后端：统一使用 9000 端口启动 FastAPI；
前端：nuxt.config.ts 中 public.apiBase = 'http://localhost:9000/api/v1'；
stores/dictionary.ts 已改成基于 apiBase 拼接（而非硬编码 8000）。
步骤 2：实现离线向量索引构建脚本
目标：从 dev.csv 构建 embeddings.npy + metadata.json。
主要子任务：
选定并安装中文向量模型依赖；
解析 dev.csv（确认字段格式和路径）；
对每条句子编码生成向量；
将向量矩阵与 metadata 写入磁盘；
在 启动.md 中记录「如何重新构建索引」命令。
产物：
scripts/build_cecsl_index.py；
data/cecsl_index/embeddings.npy；
data/cecsl_index/metadata.json。
步骤 3：在 FastAPI 中集成向量检索与随机推荐 API
目标：对外提供 /api/v1/video_rag/search 与 /api/v1/video_rag/random。
主要子任务：
在 app/ 下创建索引加载与检索模块（例如 rag_index.py）；
在 app.main 中引入并在应用启动时加载索引；
暴露两个路由：
GET /video_rag/search：参数 query, top_k；
GET /video_rag/random：参数 limit；
统一响应格式：保持与你现有 dictionary API 的风格一致（code, message, data）。
产物：
新的 Python 模块（如 app/rag_index.py）；
更新后的 app/main.py 中的路由定义。
步骤 4：前端新增视频检索 Store
目标：在前端有一个专门管理检索结果和状态的 store。
主要子任务：
新建 stores/videoRag.ts（或类似命名）：
定义 VideoResult 类型；
state：results, randomSamples, query, loading, error, stats；
actions：searchVideos, fetchRandomSamples 调用刚才的两个 API。
与现有 runtimeConfig.public.apiBase 对齐，拼接 /video_rag/... 路径。
步骤 5：在词典页面增加视频推荐区域与检索交互
目标：实现从前端完成 end-to-end 的检索与展示。
主要子任务：
修改 pages/dictionary.vue：
增加右侧布局（或下方区域），专门展示视频检索结果；
增加「句子检索」输入框与按钮，绑定到 videoRagStore.searchVideos；
页面 mounted 时调用 videoRagStore.fetchRandomSamples；
用列表展示视频缩略图/播放器 + 句子文本 + 相似度条/标签。
处理 loading / error / 空结果状态。
步骤 6：与词典的联动和 UI 打磨（可选进阶）
目标：让检索模块和现有词典自然融合。
主要子任务：
在词条详情卡片中加入「查看相关例句视频」按钮，调用视频检索；
适当优化 UI（卡片样式、颜色、响应式布局）；
如有需要，增加简单的「高/中/低匹配」文案和颜色规则。
